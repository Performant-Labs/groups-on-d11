## Summary
The plan extends every established seam (`DoShowcaseHooks`, `VariantSwitcher::build('directory.layout', …)`, `do_showcase.switcher.js`, `ShowcaseCatalog::entries()`, `views.view.all_groups.yml`) rather than forking parallel objects, and its cache-context contract (`url.query_args:variant`) is already carried by `VariantSwitcher::build()` itself — a no-op that F still needs on the outer view render for double-safety. No architectural blockers. Advisories below are non-blocking notes that will save F time.

## Findings

| # | Severity | Plan element | Drift dimension | Finding | Suggested fix |
|---|---|---|---|---|---|
| 1 | advisory | `#[Hook('views_pre_render')]` seam | layering / cache metadata timing | `views_pre_render` fires AFTER the query has run and rows are loaded — correct for injecting a `#header` render array and setting wrapper `#attributes`, but too late to add a Views field (member count) if F takes the "add Views field via YAML" path. Since the brief already keeps field additions in `views.view.all_groups.yml` (config-time), this is fine; F should NOT try to `$view->field[...] = …` inside `viewsPreRender()`. `hook_views_pre_view` (pre-query) is the wrong seam here (no rows to header-decorate yet in a useful way); `hook_preprocess_views_view` would also work but loses easy access to `$view->element['#cache']` and the `ViewExecutable` API. `views_pre_render` is the right choice — same seam `DoStreamsHooks::viewsPostRender()` / `DoGroupPinHooks::viewsPostRender()` use for view-level attach + decorate work in this codebase. Also: `VariantSwitcher::build()` already declares `#cache['contexts'] => ['url.query_args:variant']`, and Drupal bubbles child render metadata upward — F should STILL explicitly declare `$view->element['#cache']['contexts'][] = 'url.query_args:variant'` on the view's own render array (as `ShowcaseController::page()` does) so the wrapper-attribute decision itself (which happens BEFORE `VariantSwitcher::build()` is called) carries the varying context. | Keep `views_pre_render`; explicitly set the cache context on `$view->element['#cache']` in addition to the switcher's own child metadata. |
| 2 | advisory | Extend `do_showcase.switcher.js` with generic wrapper-mirror callback (per O resolution #1) | pattern consistency / single-responsibility | O decided this over a CSS `:has()` trick or a separate `do_showcase.directory-compact.js` behavior. Grep confirms this codebase has **zero existing CustomEvent/dispatchEvent patterns** — there is no established "switcher emits event, consumer listens" idiom to reuse. The extension shape must therefore stay generic and data-driven (not a directory-specific branch inside `select()`): read a `data-do-showcase-mirror-attribute` + `data-do-showcase-mirror-selector` off the radiogroup wrapper, mirror the selected `data-do-showcase-id` onto the target's attribute. That keeps the switcher single-responsibility (still owns selection + persistence), makes future instances (SC-4/SC-6/ST-8) reuse it without further JS edits, and matches how `VariantSwitcher::build()` itself is data-driven (caller passes `instance_id`/options; the widget is agnostic to what its selection means). | F: express the wiring as a generic wrapper-mirror driven by data-* attributes on the radiogroup element (set from `viewsPreRender()`), not a directory-specific `if (instanceId === 'directory.layout')` branch. |
| 3 | advisory | Shared sessionStorage key across `/showcase` and `/all-groups` | cross-cutting concerns | Confirmed safe. `/showcase`'s stub calls `$this->switcher->build('directory.layout', …, $variant)` with `$variant = $request->query->get('variant') ?? 'cards'` — the stored value is only overwritten when the user actively clicks/keys a selection (persist=true), NOT on page-load reselection (persist=false at line 145 of `do_showcase.switcher.js`). Cross-page persistence from `/showcase` → `/all-groups` is intended UX per wireframe §"Interaction with the SC-F1 `/showcase` stub" and D's decision journal. No regression path. | None. |
| 4 | advisory | `directory-presentation` catalog entry `route` = `view.all_groups.page_1` | naming / route id | Verified: `docs/groups/modules/do_chrome/tests/src/Kernel/PageHelpRouteMapTest.php:46` and `PageHelp.php:72` both key on `'view.all_groups.page_1'` — this IS the correct Views auto-generated route id for `all_groups.page_1` (path `/all-groups`). Brief's "F to verify" note is satisfied here; F can flip with confidence. | F flips `directory-presentation` to `status: 'live'`, `route: 'view.all_groups.page_1'`. |
| 5 | advisory | Fields rendered in both variants, CSS hides in each | pattern consistency | Grepped for any existing `preprocess_views_view_fields__all_groups()` implementation — **none exists**. The only reference is a docblock in `HelpText.php:239` mentioning a hypothetical `groups_chrome_preprocess_views_view_fields__all_groups()` that is not actually implemented anywhere in the tree today. No conflict. If F adds one (e.g. to compute a member-count field), it should live in `DoShowcaseHooks` (this story's owned module) rather than `do_chrome`, to keep this story's changes inside its declared "also touched" scope. | If member-count needs preprocessing, add to `DoShowcaseHooks`, not `do_chrome`. |
| 6 | advisory | Forward-compat with #125 (SC-6 map append to `views.view.all_groups.yml`) | layering | The plan leaves a clean seam: the switcher's three-option list (compact/cards/map(unavailable)) is built inside `viewsPreRender()` as a plain PHP array; #125 flips `available: false` → `true` on the map entry there. If #125 needs to add a second Views display for the map, appending a `page_2` display to the YAML does not collide with anything this story adds to `page_1`. | None. |
| 7 | advisory | Anti-duplication risk against `ShowcaseController::page()` | pattern consistency | The `viewsPreRender()` will encode roughly the same three-option array literal the controller does. Consider extracting a shared `VariantSwitcher::directoryLayoutOptions()` helper (or a constant on `VariantSwitcher`) so `/showcase` and `/all-groups` share ONE definition of the option list rather than two literals that could drift when #125 flips the map option. Non-blocking — literals in two spots is acceptable if F prefers to keep `VariantSwitcher` a pure builder; but O should be aware #125 will need to touch BOTH sites if this is not consolidated. | Optional: hoist the three-option literal to a shared source (const/static method) so #125 flips it in one place. |
