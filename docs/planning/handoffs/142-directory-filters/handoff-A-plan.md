# Handoff-A: Phase 3 — #142 MC-3 Directory Location + Language filters (up-front plan review)

**Date:** 2026-07-23
**Branch:** 142-directory-filters
**Brief reviewed:** docs/planning/handoffs/142-directory-filters/brief.md
**Reuse map:** docs/planning/handoffs/142-directory-filters/survey.md (Reuse & Analogous-Feature map section)
**Wireframe:** N/A (no new UI surface)
**Verdict:** BLOCK

## Summary
The extend-vs-new call for the *view* (add exposed filters to `views.view.all_groups.yml` rather than fork a new view) is correct and matches the existing `search`-on-label pattern. Two blocking issues break the field-side of the plan: (1) the proposed `field_group_location` string field collides on filename **and** semantics with #125 SC-6, which explicitly owns those two YAML files as a `geofield` — this is called out in #125 as "the epic's one field/schema addition." (2) The plan's language filter targets `field_group_language`, but #139 MC-4 (which #142 is coordinated with per its own issue text) owns a NEW `field_group_primary_language` and says "verify vs `do_group_language`; reuse if it already provides this." The plan must reconcile which field is authoritative before T writes tests against a name that will be renamed under it. Two smaller warns follow.

## Findings

| # | Severity | Plan element | Drift dimension | Finding | Suggested fix |

| 1 | block | New `field.storage.group.field_group_location.yml` + `field.field.group.community_group.field_group_location.yml` as string field | naming / cross-story ownership collision | #125 SC-6 explicitly owns these two filenames and defines them as `type: geofield`, flagged as "the epic's one field/schema addition." Creating them here as `type: string` produces (a) a guaranteed merge conflict on identical filenames, (b) a semantic collision (same field name, incompatible field type — the second module to install will fail schema), and (c) an unreviewed second schema addition in an epic that budgeted for exactly one. This is a "parallel path" that undoes #125's design decision. | Choose one of two paths, then update brief + survey: **(A)** Rename this story's field to something distinct and free-text-scoped, e.g. `field_group_location_text` (or `field_group_location_label`). Keep filter operator `contains`. Leaves #125's `field_group_location` (geofield) untouched. **(B)** Defer the field creation entirely and hard-depend on #125 landing first, then filter on `field_group_location`'s geofield text/place-name subproperty (contains). Option A is the lower-risk POC path and keeps stories independently mergeable. Option B changes the merge order in the issue body (currently "#124 → #125 → this OR filter-only") to a strict "must land after #125." |
| 2 | block | Language filter target = `field_group_language` (existing) | pattern consistency / dependency mismatch | #142's issue body says "coordinate with the multilingual baseline issue #139 which adds the field," and #139's Owns list is `field_group_primary_language` with note "verify vs `do_group_language`; reuse if it already provides this." Survey claims "no #139 blocker" because `field_group_language` is already on origin/main, but that field predates #139's decision on the authoritative primary-language field. Filtering on the soon-to-be-legacy name means T writes a test that either (a) breaks when #139 renames/consolidates, or (b) locks in the old name and forces #139 to keep both. | O must decide with the human: is `field_group_language` the authoritative primary-language field going forward (in which case #139 must be updated to reuse it, and #139's `field_group_primary_language` line becomes a rename)? Recommended: adopt existing `field_group_language`, add a decision-journal note obligating #139 to reuse rather than introduce a parallel field. |
| 3 | warn | Views filter `plugin_id` for the language field | Drupal-specific pitfall | The existing field is `type: language` (core Language field). The correct Views filter for that storage is `plugin_id: language` (Drupal's Language filter, offers active/current/site-default pseudo-options and a list of installed langcodes), not `plugin_id: list_field` or `plugin_id: in_operator` on a value column. The brief says "select against existing field_group_language" but does not pin the plugin_id. If F picks `in_operator` on the raw `langcode` column, the exposed widget will list every langcode string with no labels and lose the "current interface language" niceties. | Brief/survey should specify `plugin_id: language`, table `group__field_group_language`, field `field_group_language_value`, `expose: true`, `reduce: false` (or true if only exposing installed langs). | 4 | warn | Acceptance criteria — access-safety test | test coverage plan | Brief's kernel test asserts "archived/unlisted/private groups are NOT in the base result set." Good, but note the current view has only `status = 1` filter; archived/unlisted/private exclusion actually comes from Group's access grants (`gnode`-style query alter on `groups_field_data`), not from a Views filter. The kernel test must run the view **as an authenticated non-member** (or anonymous), not as UID 1, or the assertion is meaningless (UID 1 bypasses all grants and would see everything). | Add explicit note to brief: kernel test must switch to a non-privileged account before executing the view; assert result set excludes an archived + an unlisted + a private group seeded in the fixture. Cite `AccessPolicyEnforcementTest.php` as the pattern. |
| 5 | warn | Form/view display edits listed in survey | scope hygiene | Survey lists `core.entity_form_display.group.community_group.default.yml` and `core.entity_view_display.group.community_group.default.yml` as touched. If #125 is going to add `field_group_location` (geofield) to those same displays, both stories will edit the same YAML blocks. If Finding 1 is resolved by renaming this story's field, this warn dissolves; otherwise it becomes a second collision surface. | Once Finding 1 is settled (rename recommended), confirm the renamed field's widget/formatter entries are additive-only and disjoint from #125's geofield entries. |
