# Handoff-A: Phase 3 — #120 SC-1 Persona Switcher (up-front plan review)

**Date:** 2026-07-22
**Branch:** 120-persona-switcher
**Brief reviewed:** `docs/planning/handoffs/120-persona-switcher/brief.md`
**Reuse map:** `docs/planning/handoffs/120-persona-switcher/survey.md`
**Wireframe:** `docs/planning/handoffs/120-persona-switcher/wireframe.md` (APPROVED)
**Verdict:** BLOCK

## Summary
Plan is broadly sound and reuses `do_showcase` conventions faithfully (page_top idiom, one-tooltip-per-widget, HelpText append-only, seed numbering). Two hard architecture problems must be resolved before T authors RED tests, plus one ambiguity around the masquerade dependency itself:

1. **The Groups-Moderate enforcement site is misidentified.** The plan targets `user.role.groups_moderate.yml`, but every access check the persona must satisfy (`ManageMembersController::access`, `RestoreGroupAccess::access`) is evaluated against **group** permissions (`administer members`, `edit group`) resolved via the sibling `group.role.community_group-groups_moderate.yml` — a scope=outsider, `admin: true` group role that the survey does not name. Editing only the user role will not make Maria-versus-Moderator access assertions pass.
2. **`ShowcaseCatalog::personas()` already exists with the exact 4-persona list** the story needs. A brand-new `PersonaRegistry` alongside it is a parallel path, not an extension. Extend `ShowcaseCatalog` (or add a lightweight `PersonaSpec` value-object that `ShowcaseCatalog::personas()` returns — one place, one list, one contract).
3. The plan enables `drupal/masquerade` but the "always full logout+login" decision bypasses every mechanism masquerade provides (uid-1 guard, allowlist, unmasquerade session). Either use masquerade's mechanism (see finding #3 below) OR drop the dependency and ship a bespoke persona-login controller. Enabling the module without using it is dead wiring.

## Test surface (informational, for T to author RED tests)

Kernel (`docs/groups/modules/do_showcase/tests/src/Kernel/`):
- `PersonaSpecTest` — asserts `ShowcaseCatalog::personas()` returns the 4 expected ids in order; each has non-empty name + description; `moderator`'s uname resolves to `groups_moderate_demo`.
- `PersonaAccessCheckTest` — instantiates the access-check service and asserts AccessResult for: uid-1 target (denied), non-allowlisted uname (denied), each of the 4 allowlisted ids (allowed), unknown persona id (denied).
- `PersonaSwitcherRenderTest` — asserts `#cache['contexts']` includes `user`, contains 4 `<option>` entries, current selection matches session state.

Functional (BrowserTestBase):
- `PersonaSwitcherDropdownTest` — anonymous sees widget; each of 4 options present with `title=` attribute non-empty; label associated via `for`/`id`.
- `PersonaBannerTest` — after switch to Elena, banner renders with `role="status"`, exact copy `"You're browsing as Elena Garcia — Member — switch back"`, a real `<a href="/persona-switch/anonymous">` link; anonymous session = no banner in DOM.
- `PersonaAccessNegativeTest` — logged in as `groups_moderate_demo`, GET `/admin/config` → 403; GET `/admin/people` → 403; GET `/admin/modules` → 403.
- `PersonaAccessPositiveTest` — logged in as `groups_moderate_demo`, GET `/group/{n}/members` on a group with a pending relationship → 200 and pending row visible; POST restore on an archived group → succeeds; logged in as Maria, edit group page 200; logged in as Elena, edit group page 403.
- `PersonaUidOneGuardTest` — POST `/persona-switch/root` (or however uid-1 is named) as anonymous → 403 (never `user_login_finalize`d).

E2E (`tests/e2e/persona-switcher.spec.ts`):
- Full switch → verify banner → click switch-back → verify no banner, for `moderator` and `maria-chen`.
- Keyboard-only: Tab to `<select>`, arrow to Maria, Enter/Tab commits, banner appears; Tab to switch-back link, Enter, banner gone.
- Visible focus ring present on `<select>`, "Go" button, switch-back link (assert non-zero outline via computed style).
