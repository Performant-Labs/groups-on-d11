Low-fidelity mockup for the new authenticated route /my-feed/events. Both displays (Upcoming events, My RSVPs) render inside ONE shell invocation, separated by plain <h2> section headers — no second shell, no second toggle.

Contract reminder (no hardcoded routes). The iCal links ARE real, resolvable hrefs (do_discovery.ical_site / .ical_user) — those are the one intentional exception to "no hardcoded hrefs," since the issue explicitly requires linking (not reimplementing) existing, stable iCal routes.

Maps to:
  • Two DISTINCT empty-copy strings (per the design mandate) — Upcoming's copy names the prerequisite (group membership) AND offers a same-page escape hatch (switch to Global) since a 0-membership user is not necessarily stuck; My RSVPs' copy explicitly points at the OTHER section on the SAME page rather than repeating the join-a-group CTA (RSVPing needs an event to exist, not a group to join per se) — no duplicate CTA button in both empty states.
  • My RSVPs empty state carries NO cta-link (unlike Upcoming) — the honest next action ("go RSVP to something above") is already visible on the same page, so a second button pointing at the same place would be redundant chrome, not a genuine affordance.
  • Both empty blocks keep their own data-testid (upcoming-events-empty / my-rsvps-empty) distinct from #109's single shared do-streams-shell-empty, since this page can have EITHER, BOTH, or NEITHER section empty independently (unlike the single-display shell, which is all-or-nothing).
  • iCal links remain visible and correctly targeted even when both sections are empty — subscribing is still meaningful (a user's calendar will pick up future events/RSVPs without a page revisit).
State 3 of 3 — Anonymous request to /my-feed/events (no rendering)

AC: GET /my-feed/events unauthenticated → 403 (or login redirect).

No shell chrome is depicted for this state. The route's access requirement (_user_is_logged_in: 'TRUE' — per ST-1's verified Drupal-11 pattern, brief.md's binding note that _role does not gate this correctly) means an anonymous visitor never reaches a rendered do_streams_shell. Two acceptable outcomes, either is fine for this route (F picks per what ST-1's route already does for consistency):
  • 403 Forbidden — Drupal's standard access-denied page, OR
  • Login redirect — anonymous visitor is redirected to /user/login?destination=/my-feed/events, landing back on this page once authenticated.
  • Functional test (MyEventsRouteTest) asserts whichever the route actually does (403 status OR a 3xx to the login route) — this wireframe does not mandate one over the other; it is an access-control decision, not a design one.
