## F — Phase 5 (implement)
- **Decided:** Created `docs/groups/config/field.field.node.forum.comment.yml` by cloning `config/sync/field.field.node.article.comment.yml` per the brief's reuse map: new v4 uuid (`7f3d9b2a-6c41-4e08-9a3f-2d5e8c1b4f6a`), `_core.default_config_hash` omitted (matches the precedent set by `docs/groups/config/field.field.node.forum.body.yml` and `field.field.node.forum.field_group_tags.yml`, neither of which carry that key), `id: node.forum.comment`, `bundle: forum`, `dependencies.config` swapped `node.type.article` → `node.type.forum`, `settings.anonymous: 0` preserved unchanged. Every other key is byte-identical to the article template (confirmed via `diff`).
- **Decided:** Did not add entity form/view display YAMLs, did not touch the seed script, did not extend to other bundles — all per the brief's explicit "Do NOT" list and A's PASS.
- **Found (blocking GREEN, not an F-scope gap):** After assembling config and running the unmodified T-authored test, both tests remained RED — but for a *different* reason than before F's change, once diagnosed. Root-caused via a throwaway diagnostic-only test copy (never committed, deleted after use) that added the two missing `setUp()` installs to isolate exactly what was missing:
  1. **AC #2 test (`testForumBundleHasCommentField`) needs an explicit `field_config` install in `setUp()`.** `GroupsKernelTestBase`/`EntityKernelTestBase` never do a generic `config/sync` or `docs/groups/config` config-import — every fixture in this test hierarchy is built via explicit `EntityStorage::create()->save()` calls (confirmed by reading `EntityFieldManager::buildBundleFieldDefinitions()`, which only surfaces a `comment`-type field once an active `field_config` entity for `node.forum.comment` exists in the test's own DB — reading/copying the YAML into `docs/groups/config/` or `config/sync/` has no effect on a Kernel test's active config store). `HotScoreForumCommentTest::setUp()` already installs `comment_type` and `field_storage_config` (bundle-agnostic) from shipped config via `FileStorage`, but never installs the `field_config` (bundle-specific) entity — this is the missing line. `IcalFeedsTest::setUp()` (lines 102-111) is the exact working precedent: it installs both `field_storage_config` AND `field_config` from shipped config for `field.date_of_event`. With this one line added (`$entity_type_manager->getStorage('field_config')->create($shipped->read('field.field.node.forum.comment'))->save();`), `testForumBundleHasCommentField` passes.
  2. **AC #3 test (`testForumNodeHotScoreIsPositiveAfterCommentAndCron`) additionally needs `\Drupal::state()->set('comment.maintain_entity_statistics', TRUE)` in `setUp()`.** Root-caused via the same diagnostic copy: even after fix (1), the hot-score assertion still failed (`0.0` not `> 0`) because `comment_entity_statistics.comment_count` was never incremented on comment save. Traced to `\Drupal\comment\CommentStatistics::update()` (core, `web/core/modules/comment/src/CommentStatistics.php:209`), which early-returns without touching `comment_entity_statistics` unless the state key `comment.maintain_entity_statistics` is truthy. That state key is normally set by `comment_install()` (`web/core/modules/comment/comment.install:30`, `hook_install()`), but `KernelTestBase::enableModules()` (which processes `static::$modules`) never invokes `hook_install()` for enabled modules — confirmed by reading `web/core/tests/Drupal/KernelTests/KernelTestBase.php:832-876`, which only rewrites `core.extension` config and rebuilds the container/module list. Confirmed empirically: a diagnostic assertion inside the (throwaway) test copy showed `state=NULL`, `stats_row=false` (no row at all, not even a zeroed one) right after `$comment->save()`. Setting the state key explicitly in `setUp()` is the missing second line.
  - **This is a test-authorship gap in `HotScoreForumCommentTest::setUp()`, not a production-code or config gap.** No file under `docs/groups/config/` or `docs/groups/modules/do_discovery/` can set Drupal core's `comment.maintain_entity_statistics` state key or install a `field_config` entity into a Kernel test's isolated DB; only the test's own `setUp()` (or a real full site install, which does run `hook_install()`) can. F's production YAML was verified correct against both gaps via the diagnostic copy (both tests passed once the copy added the two missing lines) before concluding this is T's fix, not F's.
  - **Not an F production-code change**, and NOT applied to the shipped test file (F does not edit T's tests per pipeline convention) — flagged in `handoff-F.md` "Tests that look wrong (for T)" for T to add in Phase 6.
- **Confirmed (no regression):** Full `do_discovery` + `do_streams` Kernel suites (53 tests, 1571 assertions) — the only 2 failures anywhere are the two known-gap `HotScoreForumCommentTest` tests; all 51 others (including `IcalFeedsTest` 5/5, matching T's own sanity-check baseline) pass unchanged.
- **Evidence:** `diff` output between the new YAML and the article template (shown in `handoff-F.md`); direct reads of `EntityFieldManager.php`, `EntityKernelTestBase.php`, `KernelTestBase.php` (`enableModules`), `comment.install`, `CommentStatistics.php`; a throwaway diagnostic test copy (assembled-tree only, deleted immediately after use, never committed) that isolated each gap one at a time; full phpunit output for both the unmodified T test (still RED, unchanged failure text/line numbers from `handoff-T-red.md`) and the diagnostic copy (GREEN once both `setUp()` lines were added).
