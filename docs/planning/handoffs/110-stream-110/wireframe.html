/my-feed — ST-1 My Feed wireframe (issue #110)

Low-fidelity mockup for the new authenticated route /my-feed. The shell chrome (.shell.do-streams-shell, nav.shell-tabs, .shell-ranking) is inherited verbatim from the approved #109 wireframe (docs/handoffs/109-do-streams-scaffold/wireframe.html) — markup, classes, and data-testids are unchanged. This document adds: (1) the populated My-Feed-scoped results state, (2) the empty state with the NEW empty_cta slot, (3) a note on the anonymous 403 (no visual), and (4) the main-nav link strip for both authenticated and anonymous users.

Contract reminder (no hardcoded routes in the shell). The shell chrome below never prints a literal href — every tab/pill carries a data-* attribute mapping back to its preprocess variable id, exactly as #109 shipped it. The ONLY place /my-feed appears in this document is (a) the illustrative browser URL bar label and (b) the nav-strip link's annotated target — never inside the shell chrome markup itself.

State 3 of 4 — Anonymous request to /my-feed (no rendering)

AC-1: GET /my-feed unauthenticated → 403 (or login redirect).

No shell chrome is depicted for this state. The route's access requirement (_role: authenticated, mirroring the view display's own authenticated role access) means an anonymous visitor never reaches a rendered do_streams_shell — the response is Drupal's standard 403 (or the site's configured login-redirect behavior, unchanged by this story). There is nothing new to wireframe here; F implements this via the route's access requirement, and T asserts the HTTP status directly (AC-1). Anonymous users keep the existing /stream (Global scope) fallback, untouched by this story.
