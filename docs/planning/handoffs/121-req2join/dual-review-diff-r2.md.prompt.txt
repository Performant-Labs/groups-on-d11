## BLOCK adjudication

### [B-1] GroupAccessHook missing user-varying cache contexts — ACCEPTED

Verified: `docs/groups/modules/do_group_membership/src/Hook/GroupAccessHook.php:97,103-105` chain only `addCacheableDependency($group)` on every `AccessResult`. The hook returns different results per user (organizer via `administer members` permission gets neutral even on invite_only; non-organizer gets forbidden), so cache MUST vary by `user.permissions`. This is a real cache-correctness bug — a plain member could get an admin's cached "neutral" result and be allowed to create a membership on an invite_only group.

**F rework required (minimal):** in `groupRelationshipCreateAccess()`, chain `->cachePerPermissions()->cachePerUser()` in addition to the existing `->addCacheableDependency($group)` on every returned `AccessResult` (including the neutral early-return cases where the group is known). Matches the exact idiom `ManageMembersController::requestJoinAccess()` already established.

### [B-2] Theme picker missing render-cache metadata — DEFERRED (WARN + follow-up)

The concern is real: the three-branch action picker in `groups_chrome_preprocess_group()` computes different `$gc['action']` per current user (member/non-member) without attaching `#cache` contexts. However:

- **This is a pre-existing #85 pattern that F extended, not one F introduced.** The two original branches (`Join group` / `Leave group`, from story #85) also lack cache metadata. F's third branch inherits the gap; it does not create it.
- **Scope discipline:** fixing the theme's cache layer here would balloon the diff (touches the whole preprocess hook, requires re-testing every group render path, and potentially the `/all-groups` directory card, which A-dup already flagged as a separate follow-up), and belongs in a chrome-scoped story that audits the theme's `#cache` posture holistically — not in a membership-model story's surgical patch.
- **Immediate risk is bounded:** Drupal's dynamic page cache keys naturally on `url.path` (and BigPipe on `user`), so anonymous-visible dead links are unlikely; the failure mode is "logged-in user A sees logged-in user B's Leave button", which requires an unusual cache-warming pattern to trigger.

**Filing as a follow-up story:** "chrome: attach cache contexts to `groups_chrome_preprocess_group()` action picker (varies per user + permissions + group visibility)." Referencing #85 (original picker) and #121 (extended it). Not blocking #121.
